## Problem Statement

Multiple teams at this hackathon are already running their AI coding agents inside Multipass VMs. That's the user population: developers whose agents have shell access to fleets of Ubuntu VMs and are pushing real config changes. When one of those agents ships a bad change, the human on-call scrambles through logs to figure out what the AI did and how to undo it. This project builds an AI war room of three named agents who investigate, diagnose, and remediate an incident caused by *another* AI's bad config push. AI cleans up after AI — for an audience whose own agents are running in Multipass right now.

## What Makes This Cool

The demo isn't a thought experiment. It's a near-future scenario for a population that already exists in the room. An AI coding agent updated nginx config across three Multipass VMs and one of them won't reload because the agent generated an invalid directive. The war room agents — Optimist, Paranoid, Decider — coordinate to find the bad change, propose a revert, and write the post-mortem. The audience watches the response unfold in a chat UI with visible handoffs between agents. The personas aren't sitcom characters; they're operational roles with distinct jobs that mirror real SRE incident response.

The pitch hook for the demo opener: "If your AI agent is running in Multipass right now, this is your next outage."

## Constraints

- **5 hours total build time** (single hackathon session)
- **Solo developer** (amylily1011)
- **Demo must be 100% deterministic** — no live API failures, no real VM dependencies
- **Stage**: Canonical AI Ubuntu Hackathon, judges familiar with Ubuntu/snap/multipass
- **Hardware**: Ubuntu 24.04 dev sandbox with Node, Python, Multipass already installed
- **LLM**: Whichever API key Amy has (Claude API or OpenAI)

## Premises

1. **Mocked Multipass is acceptable AND disclosed.** The README and demo opener explicitly say "simulated Multipass incident environment for demo reliability." Honesty about the simulation removes any "is this real?" doubt from judges and keeps focus on the agent behavior.
2. **Personas are operational roles, not sitcom characters.** Each agent has a defined function in the response loop, not just a voice. Personality emerges from the role, not from jokes layered on top.
3. **UI > clever orchestration.** Judges experience named agents handing off work in a war room layout. Visible handoffs beat invisible cleverness.
4. **The incident is AI-caused.** This is the differentiator. A coding agent pushed a bad nginx directive across three VMs. Our war room cleans up after it. Ties directly to the AI Ubuntu Hackathon theme.
5. Phase 3.5 (cross-model second opinion) skipped — hackathon time pressure, not a startup decision.

## Approaches Considered

### Approach A: Streamlit + Swarm + Archetype Agents (RECOMMENDED)
- **Stack**: OpenAI Swarm (~50 lines orchestration), Streamlit `st.chat_message` UI, Python dict for mock Multipass tool outputs
- **Effort**: M (tight but doable in 5h)
- **Risk**: Low — every component is battle-tested for hackathons
- **Pros**:
  - Swarm is the simplest multi-agent framework (single file, no DAG configuration)
  - Streamlit chat UI is 30 lines, supports per-agent avatars/colors out of the box
  - Mock tools = zero external dependencies during demo
- **Cons**:
  - Streamlit looks like Streamlit — not a custom-designed app
  - Swarm is OpenAI-flavored; if using Claude API, may need a thin adapter
- **Reuses**: OpenAI Swarm examples, Streamlit chat tutorial, fake `multipass list` output

### Approach B: Custom Python orchestration + single HTML output
- **Stack**: Write the orchestration loop by hand, dump the final transcript to a static HTML page
- **Effort**: S
- **Risk**: Med — no live UI means the demo is the final artifact, not the process
- **Pros**: Ships fastest, fewest moving parts
- **Cons**: Loses the live "agents thinking" magic; reads as "I generated a report"
- **Reuses**: Plain Python + Jinja2 templates

### Approach C: Fork a CrewAI demo and reskin
- **Stack**: Find an existing CrewAI multi-agent example, swap personas and tools
- **Effort**: S
- **Risk**: Med — inheriting someone else's architectural choices, harder to debug
- **Pros**: Fastest to first running demo (potentially 1 hour)
- **Cons**: You don't own the code, judges may recognize the base template, less impressive in code-review portion of judging

## Recommended Approach

**Approach A — Streamlit + Swarm + operationally-defined agents.**

It's the sweet spot: live UI (the war room IS the demo), original code (you own it), and the AI-cleans-up-after-AI angle is uniquely yours. Swarm's handoff primitive maps to defined operational handoffs between roles.

### Agent roles

| Agent | Operational function | Voice |
|-------|---------------------|-------|
| **Optimist** | Fast recovery and common fixes. Proposes the highest-probability revert or restart first to minimize MTTR. | Pragmatic, action-biased. "Let's try X first, it's a 5-second test." |
| **Paranoid** | Evidence gathering and risk assessment. Won't act without checking the journal, the diff, and the blast radius. | Careful, evidence-citing. "Before we touch it, what does the data say?" |
| **Decider** | Incident commander. Holds the timeline, synthesizes the other two, makes the call, writes the post-mortem. | Terse, decisive. "Optimist: try X. Paranoid: verify after." |

The personas are operational roles first, voices second. The Optimist isn't a comedian — it's the fast-recovery function. The Paranoid isn't an obstacle — it's the evidence function. Both report to the Decider, who synthesizes and commits.

### The incident (AI-caused)

A coding agent ran a config-update task across `vm-web-01`, `vm-web-02`, `vm-web-03`. It generated an invalid nginx directive (`http2_max_concurrent_streams 999999;` — wrong syntax) that triggered the failure condition only on vm-web-01. vm-web-02 and vm-web-03 reloaded successfully. The war room's job: identify the AI-authored change, locate the bad directive, revert.

### Demo arc (60 seconds)

```
0:00  Alert fires in UI: "vm-web-01 nginx reload failed"
       Banner: "Simulated Multipass incident environment — demo build"
0:05  Decider opens the war room: "Three web VMs, one failed reload. Sources?"
0:10  Optimist: "Probably the same config push that hit the others. Try a
       reload retry first?"
0:15  Paranoid: "Hold. vm-web-02 and 03 reloaded clean. What's different
       about 01? Pulling the journal."
       [Tool call: multipass exec vm-web-01 -- journalctl -u nginx -n 20]
0:25  Mock output: "nginx: [emerg] invalid parameter 'http2_max_concurrent_streams'"
0:30  Paranoid: "Config diff says this was applied 14 minutes ago by
       deploy-agent. Last good revision is HEAD~1."
0:35  Optimist: "Revert is safe — vm-web-02 and 03 are on the same baseline."
0:40  Decider: "Revert vm-web-01 to HEAD~1. Open a ticket against deploy-agent
       for invalid directive generation."
0:45  [Tool call: multipass exec vm-web-01 -- nginx-config revert HEAD~1]
       [Tool call: multipass exec vm-web-01 -- systemctl reload nginx]
0:50  Status flips green
0:55  Decider writes 3-line post-mortem: cause, action, follow-up
1:00  END — Decider closes: "Filed under: AI-authored change, needs review gate"
```

The closing line is the demo's mic drop. Judges hear it as a real takeaway: AI-generated infrastructure changes need a review gate. The war room solved this incident; the bigger question is how to prevent the next one.

## Success Criteria

1. The 60-second demo runs end-to-end without intervention
2. The UI displays "Simulated Multipass incident environment" so judges know the mocking is intentional, not hidden
3. Judges can tell the three agents apart by role (not just voice) within the first 15 seconds
4. The mocked tool calls feel like real `multipass exec` output (formatting, line breaks, timestamps)
5. The closing line — AI-authored change, needs review gate — lands as a takeaway, not a throwaway
6. The post-mortem written by the Decider is good enough that someone could actually use it
7. A backup recording exists in case live demo fails
